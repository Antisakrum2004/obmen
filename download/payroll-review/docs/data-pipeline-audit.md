## Обнаруженные ошибки (10 критических)

### BUG #1: Elapsed БЕЗ пагинации — КРИТИЧЕСКАЯ ПОТЕРЯ ДАННЫХ

**Файл:** `mock-data.js:440` (старый `PR_loadRealData`)  
**Симптом:** Загружается только первая страница (50 записей) elapsed для каждой задачи  
**Влияние:** Если задача имеет более 50 elapsed записей, остальные ТЕРЯЮТСЯ бесследно

**Было:**
```javascript
bxPost('task.elapseditem.getlist', { TASK_ID: parseInt(tid) })
```
Один запрос — первая страница. Нет цикла пагинации.

**Стало:** `_prLoadDevElapsed()` — загружает ВСЕ страницы через `_prFetchAllPages()`:
```javascript
while (typeof next === 'number' && next > 0) {
  start = next;
  // следующая страница
}
```

### BUG #2: Фильтр задач по CREATED_DATE — НЕ НАХОДИТ задачи из предыдущих периодов

**Файл:** `mock-data.js:381-407` (старый `PR_loadRealData`)  
**Симптом:** Задачи, созданные ДО текущего периода, но с elapsed В текущем периоде, НЕ НАХОДЯТСЯ  
**Влияние:** Разработчик работал над задачей в мае, но задача создана в апреле — задача НЕ ПОЯВИТСЯ в обзоре

**Было:** 2 запроса:
1. `CREATED_DATE >= from AND CLOSED_DATE <= to` — только задачи созданные И закрытые в периоде
2. `CREATED_DATE >= from AND STATUS = '3'` — только задачи в работе созданные в периоде

**Пропущено:**
- Задачи созданные ДО периода, но закрытые В периоде
- Задачи без CLOSED_DATE, созданные до периода

**Стало:** 3 запроса на каждого разработчика:
1. Созданные в периоде
2. В работе (STATUS=3) — без ограничения по дате создания
3. Закрытые в периоде — без ограничения по дате создания

### BUG #3: Отсутствуют `fromStr`/`toStr` в данных — нормализация НЕ фильтрует по периоду

**Файл:** `mock-data.js:363-371`, `payroll-normalizer.js:210-216`  
**Симптом:** `buildReviewRows()` ожидает `data.fromStr` и `data.toStr`, но их НЕТ  
**Влияние:** Фильтр по периоду в `normalizeElapsedBatch` ВСЕГДА пропускается

**Было:**
```javascript
return { elapsed, tasks, projects, tasksMeta, from, to, days };
// fromStr и toStr — отсутствуют!
```

**Стало:**
```javascript
return { elapsed, tasks, projects, tasksMeta, from, to, fromStr, toStr, days };
```

Теперь `normalizeElapsedBatch` корректно фильтрует записи по дате.

### BUG #4: Условие пагинации `r.next && tasks.length >= 50` — избыточное ограничение

**Файл:** `core.js:291`  
**Симптом:** Если последняя страница содержит ровно 50 записей, но `r.next` установлен, пагинация продолжится. Но если ответ содержит менее 50 записей при наличии `r.next` (нестандартный, но возможный ответ API), пагинация ПРЕРВЁТСЯ  
**Влияние:** Потенциальная потеря страниц данных

**Было:**
```javascript
if (r && r.next && tasks.length >= 50) { start = r.next; return step(); }
```

**Стало:**
```javascript
if (r && typeof r.next === 'number' && r.next > 0) { start = r.next; return step(); }
```

### BUG #5: Нет retry логики — упавший запрос = тихая потеря данных

**Файл:** `core.js:252-263` (bxPost)  
**Симптом:** Любая сетевая ошибка или 503 от Bitrix24 → `{error: message}` → пустой результат  
**Влияние:** Временные сбои API приводят к неполным данным без предупреждения

**Стало:** `_prSafeBxPost()` — 3 повторные попытки с задержкой 1000мс для retryable ошибок (rate limit, timeout, 503, flood control)

### BUG #6: Race condition при смене периода во время загрузки

**Файл:** `tab-payroll-review.js:97-152`  
**Симптом:** Пользователь переключает месяц → начинается новая загрузка → старая загрузка завершается → рендерятся данные СТАРОГО периода с заголовком НОВОГО  
**Влияние:** Временное отображение неверных данных

**Стало:** `loadId` счётчик — каждый вызов `_prLoadData()` инкрементирует `loadId`, и при получении ответа проверяется что `loadId` не изменился. Устаревшие ответы игнорируются.

### BUG #7: N+1 запросов для elapsed — flood control риск

**Файл:** `mock-data.js:439-451`  
**Симптом:** Один API-запрос на каждую задачу для загрузки elapsed  
**Влияние:** При 500 задачах = 500 API запросов → гарантированный flood control от Bitrix24

**Стало:** Загрузка elapsed ПО РАЗРАБОТЧИКАМ — 11 запросов (по одному на разработчика) вместо N запросов (по одному на задачу). Throttle 200мс между запросами.

### BUG #8: Нет loading state machine

**Файл:** `tab-payroll-review.js`  
**Симптом:** UI показывает либо "Загрузка..." либо полную таблицу — нет различия между partial и complete  
**Влияние:** Неполный dataset отображается как полный

**Стало:** `loadState` машина состояний: `idle → loading → partial → complete` (или `failed`). Debug panel показывает текущее состояние.

### BUG #9: `PR_FORCE_MOCK = true` — сбрасывает режим на каждой перезагрузке

**Файл:** `core.js:180`  
**Симптом:** Пользователь переключился в ЖИВОЙ режим → обновил страницу → снова МОК  
**Влияние:** Невозможно работать с реальными данными стабильно

**Стало:** `PR_FORCE_MOCK` читается из `localStorage('pr_mock_mode')`. Выбор пользователя сохраняется между сессиями.

### BUG #10: Нет диагностики — невозможно понять что загружено

**Симптом:** Нет способа узнать сколько задач/elapsed загружено, были ли ошибки, сколько запросов  
**Влияние:** Невозможно отладить проблемы с данными

**Стало:**
- `window.__PAYROLL_DEBUG()` — полный отчёт в консоли
- `PR_DataLoadReport` — детальная статистика загрузки
- Расширенный debug panel в UI (всегда видимый, не только в МОК)

## Архитектура Safe Fetch Pipeline

### Новый файл: `payroll-data-loader.js`

```
LOAD TASKS (3 фильтра × 11 разработчиков)
    ↓
VERIFY PAGINATION COMPLETE
    ↓
LOAD ELAPSED (11 запросов по разработчикам, с пагинацией)
    ↓
VERIFY ELAPSED COMPLETE
    ↓
FILTER BY PERIOD + EXCLUDE GROUPS
    ↓
DETECT ORPHANS + DUPLICATES
    ↓
LOAD PROJECTS
    ↓
BUILD RESULT (with fromStr/toStr)
    ↓
VALIDATE (PR_ValidateLoadedData)
    ↓
RENDER UI
```

### Ключевые компоненты

| Компонент | Назначение |
|-----------|-----------|
| `_prSafeBxPost()` | bxPost с retry (3 попытки, задержка 1000мс) |
| `_prFetchAllPages()` | Пагинация до полного завершения (while next > 0) |
| `_prLoadDevTasks()` | Загрузка задач для разработчика (3 фильтра) |
| `_prLoadDevElapsed()` | Загрузка elapsed для разработчика (с пагинацией) |
| `PR_SafeLoadPeriodData()` | Основной pipeline — оркестрация загрузки |
| `PR_ValidateLoadedData()` | Проверка целостности загруженных данных |
| `PR_GetDebugInfo()` | Полный диагностика для `window.__PAYROLL_DEBUG()` |
| `PR_DataLoadReport` | Глобальный объект статистики загрузки |

### Защита от race conditions

Каждый вызов `_prLoadData()` получает уникальный `loadId`. При получении ответа проверяется что `loadId` совпадает с текущим. Устаревшие ответы отклоняются.

### Защита от бесконечного цикла пагинации

1. Лимит страниц: `maxPages = 100`
2. Защита от stuck: если `start` не изменился между итерациями — break
3. Проверка total: если `allItems.length < total` — предупреждение
