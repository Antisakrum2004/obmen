# Аудит рефакторинга системы зарплатного обзора

**Проект:** Зарплатный обзор (Payroll Review) для 1С-АйтиЛаб  
**Версия кода:** ПР-0.4.1  
**Дата аудита:** 2026-03-04  
**Автор:** Технический аудит

---

## 1. Что уже реализовано

Система зарплатного обзора представляет собой одностраничное веб-приложение (SPA), работающее без сборщика, на чистом ES5-JavaScript, и развёрнутое на Vercel через Flask serverless-функцию. На момент аудита реализован следующий функционал.

**Модель данных.** Введена задача-центричная модель (TaskReview), где ключом выступает пара `taskId + userId`. Для каждой записи ревью определены три типа часов: **Факт** (factHours, только чтение, из Bitrix24), **Оплачиваемые клиенту** (billableHours, редактируемые) и **К выплате** (payrollHours, редактируемые). Это позволяет менеджеру корректировать фактически отработанные часы отдельно для биллинга и отдельно для расчёта зарплаты.

**Слой нормализации.** Функция `normalizeElapsed()` в `payroll-review-calc.js` обеспечивает приведение сырых записей elapsed из Bitrix24 к единому формату, включая конвертацию секунд в минуты и часы с одним знаком после точки. Функция `groupElapsedByTask()` группирует нормализованные записи по ключу `taskId_userId` с защитой от дубликатов через `seen`-map по elapsed ID. Функция `aggregateTaskHours()` выполняет простую агрегацию минут и часов для группы.

**Загрузка данных.** Реализован двухрежимный механизм: мок-режим (`PR_MOCK_MODE = true`) генерирует реалистичные данные (41 задача, 69 записей elapsed), а живой режим (`PR_MOCK_MODE = false`) загружает данные через Bitrix24 REST API с пагинацией. Переключатель выведен в UI кнопкой МОК/ЖИВОЙ.

**Хранение.** Реализован слой `payroll-review-storage.js` на базе localStorage с префиксом `pr_` и версионированием (`PR_STORAGE_VERSION = 2`). Хранятся: ревью по периодам, настройки разработчиков (ставка, базовая часть, ИНН), фильтры, общие настройки. Предусмотрены функции очистки и получения списка сохранённых периодов.

**Интерфейс.** Полностью функциональный UI: KPI-карточки (факт-часы, оплата клиенту, к выплате, сумма выплат), таблица обзора с инлайн-редактированием billable/payroll часов, прогноз выплат по разработчикам, фильтрация по разработчику/проекту/статусу, выбор периода (6 месяцев), админ-модалка для настроек разработчиков, панель сохранения с индикатором dirty-состояния, отладочная панель в мок-режиме.

**Статусная модель.** Реализован поток статусов: `pending` → `approved` → `disputed` → `excluded` → `pending` (циклический). Статус переключается кликом по бейджу в таблице. Массовая операция «Подтвердить все» переводит все `pending` в `approved`.

**Экспорт.** Генерация CSV-файла с группировкой по разработчику, BOM для корректного отображения в Excel, разделитель «;», формат чисел с запятой. Имя файла: `зарплата_YYYY-MM.csv`.

## 2. Сильные стороны

Несмотря на раннюю стадию проекта (v0.4.1), кодовая база обладает рядом архитектурных преимуществ, которые существенно облегчат рефакторинг.

**Задача-центричная модель данных.** Это фундаментальное правильное решение. Данные уже структурированы вокруг сущности «задача + разработчик», а не вокруг месяца. Это значит, что при рефакторинге не потребуется перестраивать доменную модель — достаточно улучшить инкапсуляцию существующей. Ключ `_reviewKey` в формате `taskId_userId` уже обеспечивает уникальность и стабильность идентификации записей.

**Разделение calc.js и tab-*.js.** Файл `payroll-review-calc.js` содержит чистые функции, не зависящие от DOM: `normalizeElapsed()`, `groupElapsedByTask()`, `aggregateTaskHours()`, `buildPayrollProjection()`, `buildPeriodTotals()`. Это критически важно — эти функции можно переносить в модульную архитектуру без изменений. Файл `tab-payroll-review.js` отвечает за UI. Разделение уже существует, хотя и не полностью (подробнее в разделе 4).

**Чистый ES5 без сборщика.** Отсутствие сборщика (webpack, vite, rollup) — это одновременно и ограничение, и преимущество. Преимущество в том, что деплой тривиален: push в GitHub → Vercel auto-deploy. Нет зависимостей от версий Node.js, нет lock-файлов, нет проблем с совместимостью плагинов. Код запускается в любом браузере без транспиляции.

**Версионирование хранилища.** Наличие `PR_STORAGE_VERSION` в localStorage позволяет бесшовно мигрировать данные при изменении формата. Если при рефакторинге изменится структура сохранённых ревью, достаточно проверить версию и выполнить миграцию. Это уже заложенная защита от потери данных.

**Дедупликация в normalizeElapsed.** Механизм `seen`-map в `groupElapsedByTask()` защищает от дублей на уровне elapsed ID. Bitrix24 API иногда возвращает дубли при пагинации, и этот механизм уже работает корректно. При рефакторинге его нужно сохранить как есть.

**Мок/Живой переключатель.** Возможность работать с моковыми данными критически важна для разработки и тестирования. Моковые данные реалистичны: 11 разработчиков, 41 задача, 69 elapsed-записей, реальные проекты из Bitrix24. Это не игрушечный набор, а приближённая к продакшену модель данных.

**Flask-прокси.** Файл `api/index.py` — простая и надёжная прокси-функция. Она раздаёт статику, проксирует API-запросы к Bitrix24, обрабатывает CORS, корректно определяет MIME-типы. Этот файл не требует изменений при рефакторинге.

## 3. Архитектурные запахи

При анализе кодовой базы выявлен ряд архитектурных запахов (architectural smells), которые при масштабировании приведут к усложнению поддержки и внесению дефектов.

**core.js — Божественный объект.** Файл `core.js` объединяет в себе шесть категорий ответственности: (1) константы приложения (`PH`, `CL`, `МЕСЯЦЫ_КР`, `МЕСЯЦЫ_ПОЛН`), (2) бизнес-константы (`DEVELOPERS`, `DEV_RATES`, `DEV_BASE`, `DEV_INN`, `PROJECTS`, `STAGE_MAP`, `EXCLUDE_GROUPS`), (3) утилиты (`fmt`, `mhm`, `esc`, `truncate`, `getFirstName`, `parseBitrixDate`), (4) API-обёртки (`bxPost`, `bxPostAsDev`, `fetchTasksPaginated`), (5) помощники периода (`prGetCurrentMonth`, `prGetMonthRange`, `prGetPeriodKey`), (6) глобальное состояние (`allData`, `prCurrentPeriod`). Дополнительно функции `prGetRate`, `prGetBase`, `prGetInn`, `prGetDevName` в core.js обращаются к `prLoadDevSettings` из storage.js, создавая обратную зависимость. При рефакторинге core.js должен быть разделён минимум на 4 модуля.

**prLoadPeriodData() в mock-data.js.** Функция `prLoadPeriodData()` (строка 320 в mock-data.js) отвечает за маршрутизацию между моковыми и реальными данными, но расположена в файле, который семантически предназначен для генерации моков. Это нарушает принцип единственной ответственности (SRP). Кроме того, `PR_MOCK_generate()` использует глобальные `PROJECTS`, `EXCLUDE_GROUPS`, `DEVELOPERS`, создавая неявные зависимости.

**Отсутствие ReviewSnapshot.** Каждый раз при загрузке периода ревью собираются заново из elapsed-записей и сохранённых корректировок. Нет понятия «снимок ревью» (ReviewSnapshot), который фиксирует состояние на момент утверждения. Это означает, что если Bitrix24 вернёт новые elapsed-записи (например, запоздалый трекинг времени), ранее утверждённые расчёты «поплывут». Необходима сущность PeriodSnapshot, фиксирующая утверждённый набор данных.

**Отсутствие конечного автомата периода.** Период не имеет состояний (draft/review/approved/locked). Это позволяет редактировать уже утверждённые данные, менять ставки задним числом, повторно утверждать. В продакшене это приведёт к расхождениям между данными в системе и фактически выплаченными суммами.

**Отсутствие аудиторского следа.** Нет логирования изменений: кто, когда и какие корректировки внёс. Поле `updatedAt` обновляется, но предыдущие значения теряются. Для финансовой системы это неприемлемо — каждый пересчёт зарплаты должен быть трассируем.

**Безопасность вебхука.** Вебхук Bitrix24 зашит в коде (строка 164 core.js: `PR_DEFAULT_HOOK`) и передаётся как URL-параметр в API-вызовах (`?hook=...`). Это означает, что вебхук виден в логах сервера, истории браузера и может быть перехвачен. Необходимо перенести аутентификацию на серверную сторону.

## 4. Смешение UI и бизнес-логики

Наиболее критичный архитектурный запах — смешение ответственности между UI-слоем (tab-payroll-review.js) и бизнес-слоем. Рассмотрим конкретные примеры.

**_prSaveAll() — конструкция бизнес-объектов в UI-коде.** Функция `_prSaveAll()` (строки 521–537 tab-payroll-review.js) напрямую конструирует объекты ревью из `_pr.rows`:

```javascript
function _prSaveAll() {
  var reviews = {};
  _pr.rows.forEach(function(r) {
    reviews[r._reviewKey] = {
      billableHours: r.billableHours,
      payrollHours: r.payrollHours,
      rate: r.rate,
      base: r.base,
      reviewStatus: r.reviewStatus,
      managerComment: r.managerComment,
      updatedAt: r.updatedAt
    };
  });
  prSaveReviews(prCurrentPeriod.year, prCurrentPeriod.month, reviews);
  _pr.dirty = false;
  _prRenderAll();
}
```

Проблема: UI-функция знает структуру бизнес-объекта (какие поля сохранять), знает ключ хранилища (`_reviewKey`), и знает, как вызвать storage-функцию. При изменении модели данных нужно менять UI-код. Правильное решение — делегировать сохранение сервисному слою: `ReviewService.saveAll(rows, period)`.

**_prSaveAdmin() — чтение DOM для бизнес-операций.** Функция `_prSaveAdmin()` (строки 572–595) читает DOM через `querySelectorAll('.pr-admin-input')`, извлекает `data-devid` и `data-field`, группирует по разработчику, конвертирует типы, и сохраняет через `prSaveDevSettings()`. Это антипаттерн «UI как источник данных». Бизнес-операция (сохранение настроек разработчика) не должна зависеть от структуры DOM. Правильный подход — передавать данные из модалки через объект, а не через DOM.

**_prOnEdit() — мутация бизнес-состояния из UI-обработчика.** Функция `_prOnEdit()` (строки 448–478) выполняет: (1) парсинг значения из input, (2) валидацию, (3) прямую мутацию `_pr.rows[realIdx]`, (4) пересчёт `payrollAmount`, (5) обновление `updatedAt`, (6) установку dirty-флага, (7) пересчёт projection и totals, (8) полный ре-рендер. Это 8 ответственностей в одной функции. Дублирование формулы расчёта: `payrollAmount = Math.round(payrollHours * rate) + base` повторена здесь и в `buildTaskReviewRows()` (строка 146 calc.js).

**buildTaskReviewRows() — зависимость чистой функции от storage.** Функция `buildTaskReviewRows()` в calc.js вызывает `prGetDevName()` и `prGetRate()`, которые внутри обращаются к `prLoadDevSettings()` из storage.js. Это нарушает заявленную чистоту calc.js. Функция, которая должна быть чистой, имеет побочный эффект — чтение из localStorage. Правильное решение — передавать настройки разработчиков как параметр.

## 5. Агрегация внутри рендеринга

Проблема совмещения агрегации данных и построения HTML в одних и тех же функциях ведёт к тому, что бизнес-логика невозможно протестировать изолированно от DOM.

**_prRenderTable() — фильтрация и рендеринг.** Функция `_prRenderTable()` (строки 231–296) вызывает `_prGetFilteredRows()`, которая фильтрует `_pr.rows` по текущим фильтрам, и затем в том же цикле строит HTML для каждой строки. Результат фильтрации нигде не кэшируется и не доступен другим функциям. Если понадобится показать количество отфильтрованных строк в заголовке — придётся дублировать вызов `_prGetFilteredRows()`.

**_prSumField() — дублирование calc.js.** Функция `_prSumField()` (строки 624–632 tab-payroll-review.js) дублирует логику агрегации из `buildPeriodTotals()` (calc.js). Разница лишь в том, что `_prSumField()` суммирует произвольное поле по отфильтрованному набору строк, исключая `excluded`. Однако `buildPeriodTotals()` делает то же самое, но по полному набору. Это не переиспользование, а параллельная реализация той же логики:

```javascript
// tab-payroll-review.js — дублированная агрегация
function _prSumField(rows, field) {
  var sum = 0;
  rows.forEach(function(r) {
    if (r.reviewStatus !== 'excluded') {
      sum += r[field] || 0;
    }
  });
  return sum;
}

// calc.js — оригинальная агрегация
rows.forEach(function(r) {
  if (r.reviewStatus !== 'excluded') {
    totals.totalFactHours += r.factHours;
    totals.totalBillable += r.billableHours;
    // ...
  }
});
```

**_prRenderKPIs() — доступ к предвычисленным данным с остаточной связностью.** KPI-карточки используют `_pr.totals`, который вычисляется в `buildPeriodTotals()`. Сам по себе доступ к предвычисленным данным — это правильно. Однако рендеринг знает структуру totals (имена полей `totalFactHours`, `totalBillable`, etc.), что создаёт жёсткую связность между calc.js и UI. Изменение формата totals сломает рендеринг.

**Последствия для тестирования.** Ни одну из функций рендеринга нельзя вызвать в Node.js-тесте без DOM (jsdom/puppeteer). Это означает, что агрегация, зашитая в рендер, тестируется только вручную. Вынос агрегации в отдельный чистый слой (например, `computeFilteredTotals(filteredRows)`) позволит покрыть тестами 80% бизнес-логики без браузера.

## 6. Жёсткая связанность хранилища

Слой хранения (`payroll-review-storage.js`) реализован исключительно на localStorage, и это создаёт множественные точки жёсткой связанности.

**Прямые вызовы localStorage из разных модулей.** Помимо storage.js, прямой доступ к localStorage осуществляется в: (1) `core.js` строка 166 — чтение `bx_hook` для инициализации HOOK; (2) `index.html` строка 138 — запись `bx_hook` через `saveH()`. Это означает, что при переходе на серверное хранение (IndexedDB, API) нужно менять не только storage.js, но и core.js, и index.html.

**core.js зависит от storage.js.** Функции `prGetRate()`, `prGetBase()`, `prGetInn()`, `prGetDevName()` в core.js вызывают `prLoadDevSettings()` из storage.js. Это создаёт циклическую семантическую зависимость: core.js загружается первым (script-тег), но его функции зависят от storage.js, который загружается позже. Работает только потому, что вызовы отложены до исполнения, но порядок загрузки скриптов становится критичным.

**calc.js зависит от storage.js.** Как отмечено выше, `buildTaskReviewRows()` вызывает `prGetDevName()` и `prGetRate()`, которые читают из localStorage. Это делает невозможным использование calc.js в изолированной среде (например, в backend-расчётах или тестах) без мокирования localStorage.

**Отсутствие уровня абстракции.** Нет интерфейса хранения, который бы позволил подставлять разные реализации. Все функции хранения возвращают конкретные типы (объекты из JSON.parse(localStorage)), и все вызывающие стороны рассчитывают на этот формат. Замена localStorage на серверный API потребует: (1) создания абстрактного слоя `IStorage` с методами `load`, `save`, `delete`; (2) двух реализаций: `LocalStorageImpl` и `ServerStorageImpl`; (3) замены всех прямых вызовов на вызовы через абстракцию; (4) добавления синхронизации для offline-режима.

**Проблема лимита localStorage.** Объём localStorage ограничен ~5 МБ. При 11 разработчиках и ~50 задачах в месяц один период занимает ~30 КБ в JSON. За 6 месяцев — ~180 КБ. С учётом настроек разработчиков и фильтров — ~200 КБ. Пока это не критично, но при масштабировании (больше разработчиков, больше задач, история за год) лимит будет достигнут. Кроме того, localStorage блокирует основной поток при чтении/записи больших объёмов.

## 7. Потенциальная порча состояния

Текущая архитектура использует изменяемые (mutable) данные повсюду, что создаёт риски порчи состояния, особенно при множественных одновременных операциях.

**_pr.rows — изменяемый массив с прямой мутацией.** Объект `_pr` (строки 6–20 tab-payroll-review.js) содержит `rows: []`, который мутируется в пяти местах: (1) `_prOnEdit()` — изменение полей строки; (2) `_prCycleStatus()` — изменение `reviewStatus`; (3) `_prSort()` — перестановка элементов через `Array.sort()`; (4) `_prApproveAll()` — массовое изменение статуса; (5) `_prLoadData()` — полная замена массива. Нет механизма отката: если пользователь изменил billable-часы и случайно нажал «Подтвердить все», отменить можно только перезагрузкой страницы (с потерей несохранённых правок).

**Мутация при сортировке.** Функция `_prSort()` (строки 501–513) вызывает `_pr.rows.sort()`, который мутирует исходный массив. Это означает, что порядок строк в `_pr.rows` зависит от последнего действия сортировки, а не от детерминированного ключа. Если два обработчика выполнят сортировку с разными полями (например, при быстрых кликах по заголовкам), порядок может стать непредсказуемым.

```javascript
function _prSort(field) {
  if (_pr.sortField === field) {
    _pr.sortDir = -_pr.sortDir;
  } else {
    _pr.sortField = field;
    _pr.sortDir = 1;
  }
  _pr.rows.sort(function(a, b) {
    var va = a[field], vb = b[field];
    if (typeof va === 'string') return va.localeCompare(vb) * _pr.sortDir;
    return (va - vb) * _pr.sortDir;
  });
  _prRenderAll();
}
```

**Глобальные изменяемые переменные.** Переменные `prCurrentPeriod`, `PR_MOCK_MODE`, `PR_FORCE_MOCK`, `HOOK`, `allData` — глобальные изменяемые переменные. Любая функция в любом файле может изменить их значение. Например, `toggleMockMode()` в index.html меняет `PR_MOCK_MODE` и `PR_FORCE_MOCK`, а `saveH()` меняет `HOOK` и `PR_MOCK_MODE`. Нет инкапсуляции, нет гарантии консистентности.

**_prOnEdit и проблема индексов.** Функция `_prOnEdit()` получает `data-idx` из DOM, что соответствует индексу в отфильтрованном массиве, а затем ищет реальный индекс через `_pr.rows.indexOf(row)`. Если между рендерингом и обработчиком пользователь отсортировал таблицу, индекс может указывать на другую строку. Хотя `indexOf` по ссылке объекта mitigates это частично, при сложных сценариях (фильтрация + сортировка + параллельные правки) возможны ошибки.

**Отсутствие иммутабельности.** Нет ни одного паттерна иммутабельных данных. Все объекты строк — прямые ссылки, которые мутируются на месте. Это делает невозможным реализацию undo/redo, сравнение снимков состояния, оптимизацию ре-рендеринга через shallow comparison. При рефакторинге необходимо ввести иммутабельные обновления: вместо `_pr.rows[idx].field = val` использовать `_pr.rows = _pr.rows.map((r, i) => i === idx ? {...r, field: val} : r)`.

## 8. Риски масштабирования

Текущая реализация оптимизирована для прототипа с 11 разработчиками и ~50 задачами в месяц. При росте нагрузки возникнут следующие проблемы.

**N+1 запросов к API.** Функция `PR_loadRealData()` (строки 375–516 mock-data.js) выполняет: (1) `DEV_IDS.length` запросов для загрузки задач по каждому разработчику (11 запросов); (2) для каждого разработчика — ещё один запрос за незакрытыми задачами (ещё 11); (3) `taskIds.length` запросов для загрузки elapsed по каждой задаче (50+ запросов). Итого: 72+ API-вызова для одного периода. При 1000 задач это будет 1022+ запросов. Bitrix24 API имеет rate limit (обычно 2 запроса/секунду на вебхук), что означает время загрузки ~8 минут для 1000 задач. Решение: батчинг elapsed-запросов, кэширование задач, параллельная загрузка с контролем concurrency.

**Отсутствие пагинации в таблице.** Все строки рендерятся за один проход. При 1000+ строк innerHTML будет содержать ~500 КБ HTML, и браузер потратит значительное время на парсинг и отрисовку. Нет виртуального скроллинга или постраничной навигации.

**Полный ре-рендер при каждом изменении.** Функция `_prRenderAll()` (строка 108) полностью заменяет `innerHTML` контейнера при любом действии: редактирование ячейки, переключение статуса, сортировка, фильтрация. При 1000 строк это означает пересоздание ~10 000 DOM-элементов на каждое нажатие клавиши (если бы был oninput вместо onchange). Даже с onchange, при массовом подтверждении (_prApproveAll) происходит полный ре-рендер, который занимает ощутимое время.

**Отсутствие debounce на инлайн-редактировании.** Хотя сейчас используется `onchange` (срабатывает при потере фокуса), при переходе на `oninput` для мгновенной обратной связи каждый нажатие клавиши вызовет пересчёт projection и totals через `buildPayrollProjection()` и `buildPeriodTotals()`, которые оба итерируют по всем строкам. Необходим debounce с задержкой 300–500 мс.

**Отсутствие кэширования API-результатов.** Каждый вызов `_prLoadData()` заново загружает все данные из Bitrix24, даже если период не изменился. Автообновление каждые 5 минут (строка 55: `setInterval(..., 300000)`) каждый раз делает полный набор запросов. Необходим InvalidationCache с TTL и возможностью принудительного обновления.

**Лимит localStorage.** Как отмечено в разделе 6, при росте данных лимит 5 МБ будет достигнут. Критический порог: ~150 периодов (12+ лет) для текущего размера данных, или ~2 года при 10x росте задач.

**Отсутствие Web Worker для вычислений.** Функции `buildTaskReviewRows()`, `buildPayrollProjection()`, `buildPeriodTotals()` выполняются в основном потоке. При 10 000+ elapsed-записей нормализация и группировка займут 50–100 мс, что вызовет заметное «зависание» UI.

## 9. Что можно безопасно переиспользовать

При рефакторинге важно не переписывать то, что уже работает корректно. Следующие компоненты могут быть перенесены в новую архитектуру без изменений или с минимальной адаптацией.

**normalizeElapsed() — чистая функция, готова к переиспользованию.** Функция принимает entry, возвращает нормализованный объект или null. Не зависит от DOM, не мутирует входные данные, не имеет побочных эффектов. Единственная зависимость — `DEV_IDS` из core.js, которую можно передать параметром. Качество реализации высокое: обрабатываются NaN, отрицательные значения, пустые taskId/userId.

**groupElapsedByTask() — чистая функция с дедупликацией.** Принимает массив elapsed-записей, возвращает словарь групп по ключу `taskId_userId`. Защита от дублей через seen-map работает корректно. Не зависит от DOM. Зависимость от `normalizeElapsed()` и `DEV_IDS` можно параметризовать.

**aggregateTaskHours() — простая чистая функция.** Принимает группу, возвращает агрегированный объект. Ноль зависимостей, ноль побочных эффектов. Переносится как есть.

**generateCSV() / downloadCSV() — чистые утилиты экспорта.** Функция `generateCSV()` принимает массив exportRows, возвращает строку CSV. Не зависит от DOM, не зависит от глобального состояния. Функция `downloadCSV()` использует DOM для создания Blob и ссылки, но это неизбежно для скачивания файла. Обе функции переносятся без изменений.

**PR_CSS — стили как JS-строка.** Файл `payroll-review-styles.js` содержит CSS в виде строки `PR_CSS`, которая вставляется через `<style>` в document.head. Это не самый элегантный подход, но он работает, и стили не нужно менять при рефакторинге бизнес-логики. При переходе на модульную систему можно вынести CSS в отдельный файл, но это косметическое изменение.

**Flask-прокси (api/index.py).** Серверная часть полностью готова к продакшену: раздача статики, проксирование API, CORS, правильные MIME-типы, catch-all для SPA-роутинга. Не требует изменений.

**vercel.json — конфигурация деплоя.** Роутинг настроен корректно: `/api/*` → Flask-функция, `/*` → Flask-функция (раздача статики + SPA fallback). Работает без изменений.

**Моковые данные (mock-data.js, функции генерации).** Структура `PR_MOCK` и функция `PR_MOCK_generate()` создают реалистичный набор данных. При рефакторинге их нужно отделить от `prLoadPeriodData()` и `PR_loadRealData()`, но саму генерацию можно сохранить как есть.

## 10. Что нельзя ломать

При рефакторинге критически важно сохранить работоспособность следующих компонентов и контрактов. Любое изменение, нарушающее эти контракты, блокирует релиз.

**Маршруты: /, /js/*, /api/*.** Vercel-деплой настроен так, что все запросы идут через Flask-функцию (api/index.py). Маршрут `/` отдаёт index.html, `/js/*` — JS-файлы, `/api/*` — прокси к Bitrix24. Изменение роутинга в vercel.json или api/index.py приведёт к неработоспособности приложения. Особенно чувствителен catch-all маршрут `/<path:filepath>`, который обеспечивает SPA-fallback.

**Структура HTML-таблицы.** Таблица обзора (`_prRenderTable()`) генерирует HTML с определённой структурой классов и data-атрибутов. Инлайн-редактирование зависит от `data-idx` и `data-field` на input-элементах. Переключение статуса зависит от `data-idx` на span.pr-status. Любое изменение структуры без соответствующего обновления обработчиков сломает интерактивность.

**Пайплайн деплоя: Vercel + GitHub auto-deploy.** Push в основную ветку автоматически триггерит деплой на Vercel. Это работает через интеграцию Vercel с GitHub. Изменение структуры проекта (перемещение api/index.py, изменение vercel.json) может сломать деплой. Любые структурные изменения нужно тестировать на ветке перед слиянием.

**Поток экспорта CSV.** Экспорт — критичный для бизнеса функционал: CSV импортируется в 1С. Формат CSV фиксирован: разделитель «;», BOM `\uFEFF`, числа с запятой вместо точки, заголовки «ФИО;ИНН;Период;Часы;Ставка;Базовая;Сумма;Комментарий». Изменение формата сломает импорт в 1С. Функция `prExportCSV()` вызывает `_prSaveAll()` перед экспортом — это нужно сохранить, чтобы экспортировались актуальные данные.

**Переключатель МОК/ЖИВОЙ.** Глобальные переменные `PR_MOCK_MODE` и `PR_FORCE_MOCK` управляют режимом загрузки данных. Функция `toggleMockMode()` в index.html переключает режим и вызывает `refresh()`. Этот механизм используется для тестирования и демонстрации. Его поломка сделает невозможной разработку без доступа к Bitrix24.

**Админ-модалка.** Модалка настроек разработчиков (`_prRenderAdminModal()` / `_prSaveAdmin()`) — единственный способ изменить ставки, базовые части и ИНН. В продакшене эти данные должны быть в 1С, но пока это единственный источник. Поломка админки = невозможность корректировать ставки.

**Система фильтров.** Фильтрация по разработчику, проекту и статусу работает через `_prGetFilteredRows()` и `_pr.filters`. Фильтры сохраняются в localStorage. Пользователи ожидают, что фильтры сохраняются между сессиями. Нарушение этого поведения будет воспринято как регрессия.

**Выбор периода.** Выпадающий список с 6 последними месяцами, изменение которого перезагружает данные. Период хранится в глобальной переменной `prCurrentPeriod`. Ревью сохраняются по периоду. Нарушение связи между периодом и ключом хранения приведёт к потере данных.

**Отображение версии.** Переменная `APP_VERSION` в core.js и элемент `#appVersion` в index.html. Версия отображается в шапке и важна для отладки. Не менять формат без согласования.

**Статусный поток.** Цикл `pending → approved → disputed → excluded` реализован в `_prCycleStatus()`. Статусы используются в: (1) рендеринге (CSS-классы `row-approved`, `row-excluded`), (2) фильтрации, (3) агрегации (исключённые не учитываются), (4) экспорте (исключённые не попадают в CSV). Добавление новых статусов или изменение порядка без обновления всех четырёх мест приведёт к несогласованности.
